# Plan: ExpressLoad file closure + Tier 2 refactor (2026-07-19)

Goal chain: **disk 30/30 logical-exact → Tier 2 refactor → public (GitHub)
push → then Rez includes.** This doc scopes the first two.

Everything here is grounded in a structural diff of the six remaining
non-exact System Disk files against their goldens (segment lists below are
measured, not assumed). Read `docs/REFACTORING.md` §0 (Prime Directive) first;
E-packets are *behavior-adding* (new bytes must match gold), R-packets are
*zero-drift* refactors.

Key observations:

1. **Tool015/016/018 are one missing feature, not three bugs.** Every
   non-JT load segment's LENGTH already matches gold exactly. The JT bytes
   themselves are already generated byte-exact (`expressload.encode_jumptable`,
   gated per-segment in toolcheck's `_link_jt_tool`). The constant Δ79 in the
   `~ExpressLoad` directory = exactly one more segment's directory entry
   (HET entry + SegHeaderEntry + conversion slot). The *file* builder path
   (`expressload.expressload()`) simply never emits the JT segment.
2. **Tool034 (TextEdit) is a genuine byte-exactness project**, not packaging:
   the built `main` memory image is 4,444 bytes short while the file is
   10,032 bytes long (reloc dictionary bloat — standalone relocs where gold
   compresses to SUPER, and/or case-B misfires). Tool034 is NOT in
   toolcheck's TOOLMAP — it has never been through the per-segment grinder
   that closed the other 11 tools.
3. **TS2/TS3 have no builders at all.** Their recipes live in
   `Patch2/ts2.makeout` / `Patch3/ts3.makeout` (TS3 re-assembles
   `WindMgr/NewCalls.asm`, which carries the `myEventFilter+$80000000`
   case-B addend — the reason the old notes said "multiseg expressload path
   emits NO standalone relocs"). They also need the `INIT` segment KIND
   (0x2000) emitted correctly.

## Phase E — ExpressLoad file closure (disk 24 → 30)

### E0 (= R9). Decompose expressload.py FIRST  *(zero-drift refactor, gate identical)*

`_build_het_lconst` (271 lines) and `expressload()` (221 lines) are about to
grow the JT-segment path; decompose them before feature work doubles their
complexity. Extract pure `bytes -> bytes` builders with golden layout offsets
in docstrings (cross-ref `docs/design/expressload.md`):
`_het_entries`, `_seg_conversion_table`, `_seg_header_block`, `_pathname_block`.
Verify: full recipe, every metric AT baseline. ~1-2 days. **Blocks E1.**

### E1. Emit the ~JumpTable segment in the full-file path → Tool015/016/018

Teach `expressload()` to accept the already-working `jt` inputs (the same
`abs_extra`/jt plumbing toolcheck's `_link_jt_tool` uses) and:

1. Insert the `~JumpTable` KIND-0x0002 load segment at its golden position
   (after the static segments, before the first dynamic one — confirm against
   all three goldens; Tool018 shows the order MAINPart, CopyBits, ~JumpTable,
   then the three dynamic segs).
2. Grow the `~ExpressLoad` directory accordingly: one HET entry, one
   SegHeaderEntry, one conversion slot (the measured Δ79). The directory
   builder from E0 makes this a table-append, not surgery.
3. Wire `work/diskbuilders/expressload_files.py` to pass the JT for the three
   tools (reuse toolcheck's per-tool jt_segments spec — do not re-derive it).

Acceptance: Tool015/016/018 full-file byte-exact in diskcheck
(logical-exact 24 → 27); toolcheck/gate unchanged everywhere else.
Risk: low — all constituent bytes already proven. ~1-2 days.

### E2. TS2/TS3 residual closure

CORRECTION (2026-07-19): the original "NO BUILDER" claim was wrong — full
builders already exist in `work/diskbuilders/toolsets.py`, transcribed from
`Patch2/ts2.makeout` / `Patch3/ts3.makeout`, wired into diskcheck. Post-E1
state (the E1 `_het_entries` entry1-size correction applies to all N≥2
multiseg callers and moved both files slightly toward gold): TS2 3037/36665
first-diff @0x5e, TS3 3898/41700 first-diff @0x6e — both lengths correct to
within a few hundred bytes and, per toolsets.py's own analysis, long aligned
stretches are byte-identical (TS3 MAIN matches through 0x2db6). The low raw
match is alignment shift after early directory/reloc deltas.

1. Localize the first diffs (both now in the `~ExpressLoad` directory area or
   the first segment header) — directory-building deltas vs the tools' now-
   proven path should be smallest first.
2. The case-B addends (TS3 = NewCalls.asm `myEventFilter+$80000000`, same
   record that closed Tool014) must produce standalone flagged relocs in the
   multiseg path — E1 added exactly this machinery for the tools (case-A
   standalone scan + INTERSEG/cINTERSEG emission); TS2/TS3 likely need it
   enabled/extended on their no-JT path.
3. Compare INIT/MAIN/BIGONLY per-segment first (kernelcheck-style), THEN the
   full file — per-segment failures are assembler/linker bugs to be
   root-caused, not packaging noise.

Acceptance: TS2/TS3 byte-exact (logical-exact 27 → 29). Risk: medium —
per-segment surprises possible (this is Tool.Setup patch code). ~2-4 days.

### E3. Tool034 / TextEdit byte-exactness

The last file, and the only one that is a real unknown. Treat it exactly like
Tool018 was treated:

1. **Map it into toolcheck** (TOOLMAP entry from its makefile; TextEdit
   sources under the toolbox tree) — this puts the 4,444-byte memory-image
   gap through the proven per-segment grinder. Expect assembler-dialect bugs
   (that is what every other tool's gap turned out to be). Each fix needs its
   corpus-free fixture, per the house rule.
2. Only after the code image is byte-exact, attack the reloc dictionary: the
   +10,032-byte file bloat means gsasm emits standalone relocs where gold
   uses SUPER records (and/or case-B misfires). Diagnose with a record-type
   histogram of gold-vs-built (`omf.parse_records` both sides), then fix the
   SUPER eligibility rule in the expressload/linkiigs path.
3. Gate: TOOLMAP mapping is gate-visible (tool_bytes denominator GROWS);
   never map it while non-exact without an explicit informational carve-out.

Acceptance: Tool034 byte-exact (logical-exact 29 → 30 = **whole disk**).
Risk: highest of the three — unknown bug surface. Budget ~1 week, may close
faster (Tool018's "unclosable" residual fell in one session).

**Do NOT start E2/E3 before E1 lands** — E1's directory-builder changes are
the foundation both build on.
