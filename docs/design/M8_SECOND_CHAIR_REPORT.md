# M8 second-chair report

Date: 2026-07-05

Scope: review `docs/design/M8_DISK_IMAGES.md` against the current `work/diskcheck.py`
harness and the existing component checkers. This is a design review, not an
implementation patch.

## Verdict

The overlay-first strategy is the right shape for M8. It preserves the hard part
of a shipping ProDOS image - directory layout, bitmap, index blocks, file order,
dates, and 2IMG wrapper bytes - while letting gsasm prove one file at a time that
its generated bytes can replace the original bytes in place.

Do not treat the current image byte-match as a sufficient Phase 2 gate yet. The
harness starts from the golden disk, so a no-builder run is expected to be 100%.
Before wiring real builders, M8 needs a stricter manifest, exact builder contract
checks, and a sparse-block assertion. Without those, the reportable percentage and
even a byte-identical image can mislead.

## Findings

### P0: Sparse blocks can mask bad builder output

`overlay()` skips data-block number 0, correctly treating it as an unallocated
sparse block. However, it does not assert that the corresponding builder bytes are
zero. If a builder emits nonzero data for a sparse logical block, the overlay skips
that block and the disk image can still remain byte-identical because no physical
bytes were changed.

This is present on the current System Disk, not theoretical:

```text
/System.Disk/System/FSTs/Pro.FST has sparse logical blocks at indexes 2 and 3
```

I verified that modifying `Pro.FST` builder content to nonzero bytes in sparse
block index 2 still leaves the reconstructed image byte-identical after overlay.

Recommendation:

- In `overlay()`, when `blk == 0`, require the corresponding content chunk to be
  all zeroes.
- For every builder, compare the builder's logical output to `vol.read_file(path)`
  before overlay. Image byte-match should be the final physical-image check, not
  the only builder correctness check.

### P1: Build/substitute classification disagrees with the M8 design

The design says these System Disk files are build-scope:

```text
GS.OS, Start.GS.OS, CDev.Data
```

The current type-driven classifier marks all three as substitute because they are
file type `$F9`.

The design says `BASIC.System` is substitute, but the classifier marks it as ours
because it is file type `$FF`.

This changes the worklist and the headline percentage. The current 81% is the
script's current classification, not the design's written scope.

Recommendation:

- Replace broad type-driven ownership with an explicit path manifest for Disk 2.
- Include expected file type, aux type, fork policy, expected EOF, owner
  (`build`, `substitute`, `future/rez`, `out-of-scope`), and optional builder.
- Treat file type sets as audit helpers only. They should not decide ownership for
  known shipping files.
- Fail the inventory if an on-disk file is not in the manifest.

### P1: The byte accounting denominator is unstable

`DiskFile.data_eof` currently uses the directory entry EOF. For extended files,
that is the extended-key EOF, often 512 bytes, not the data fork or resource fork
size.

Measured on the current System Disk:

```text
current entry-EOF total:          496145
actual data-fork total:           662792
actual data+resource-fork total:  769794

/System.Disk/System/Start:
  entry_eof: 512
  data_eof:  146924
  rsrc_eof:   52395
```

The 81% source-buildable figure is useful as a current script output, but it is
not a durable project metric. It undercounts substituted forked files and does not
make clear whether resource forks are in the denominator.

Recommendation:

- Report three separate numbers:
  - logical data-fork bytes built from source
  - resource-fork bytes built from source or substituted
  - physical disk-image bytes that differ/match
- Compute fork EOFs via `_resolve_fork()` for extended files.
- Keep the ROM-style headline honest: "N% of logical fork bytes built" and
  "100% byte-identical disk image" are different claims.

### P1: Existing checkers are not yet disk-file builders

M8 correctly states that `SOURCE_BUILDERS` must return the full on-disk file, not
the de-ExpressLoaded code image. Most existing component checkers are comparison
harnesses, not packaging builders:

- `toolcheck.py` returns de-ExpressLoaded code images.
- `fstcheck.py` and `drivercheck.py` compare code images for ExpressLoaded files.
- `kernelcheck.py` currently excludes `Loader.bin` from `GS.OS`, compares
  `Start.GS.OS` imperfectly, and compares only the PROCONE slice of `P8`.

Recommendation:

- Add a small builder API whose return value is always exact disk-file bytes:
  `build_disk_file(name) -> bytes`.
- For ExpressLoaded tools/FSTs/drivers, the builder must run link plus
  `gsasm.expressload.expressload(...)`, then compare the full file to the disk
  file before overlay.
- For MakeBin/catenate outputs, compare the full produced file bytes to the
  extracted disk file, not only a sliced or de-ExpressLoaded image.
- Do not wire a checker function directly into `SOURCE_BUILDERS` unless it already
  returns full disk-file bytes and exact EOF length.

### P2: A 100% image match with zero builders is expected

`diskcheck.py` starts from the original `.2mg` and overlays only wired builders.
With `SOURCE_BUILDERS` empty, the reconstructed image is the original image.
That is a useful round-trip smoke test, but it is not a proof of source rebuild.

Recommendation:

- Add an explicit CI mode, for example `--require-built N` or
  `--min-built-bytes N`, so coverage cannot silently drop to zero.
- Track built-byte coverage as a monotonic metric once Phase 2 starts.
- Print and gate "builder exactness" separately from "image byte-match".

### P2: Phase 2 should start narrower than the prose suggests

The prose says to start with "already byte-exact" files. Based on the current
harnesses, the safest first disk builders are:

```text
ProDOS      full file exact today
Error.Msg   code image exact; needs full OMF file builder check
```

Next candidates, after full-file packaging adapters exist:

```text
Char.FST
Console.Driver
Tool014
Tool021
Tool022
Tool028
```

Do not count these as disk-ready until their builders emit full on-disk files and
pass the logical file compare. `GS.OS`, `Start.GS.OS`, `P8`, `Pro.FST`,
`AppleDisk3.5`, and `AppleDisk5.25` should remain residual worklist items.

### P3: `a2til` is hard-coded by absolute path

`work/diskcheck.py` imports `/Users/mdj/src/a2til` directly. That is acceptable for
local exploration, but it makes the M8 harness machine-specific.

Recommendation:

- Resolve `a2til` through an environment variable, editable config, sibling repo
  discovery, or a submodule path.
- Fail with a clear message when it is missing.

## Recommended Phase 2 gate

For each manifest entry with a builder:

1. Build full disk-file bytes.
2. Assert `len(content) == data_fork_eof`.
3. Assert every sparse logical block in `content` is zero.
4. Assert `content == vol.read_file(path)` before overlay.
5. Overlay into the original data blocks.
6. Assert the full image is byte-identical.
7. Increase or preserve the built-byte coverage metric.

This keeps the M8 discipline tight: logical file correctness first, physical disk
identity second, coverage accounting third.

## Suggested next work

1. Land the explicit Disk 2 manifest and fix the classification mismatches.
2. Harden `overlay()` with exact EOF and sparse-zero assertions.
3. Recompute fork-aware coverage metrics.
4. Wire only `ProDOS` first, using `work/probootcheck.py`'s full-file builder path.
5. Add `Error.Msg` once its builder returns the full on-disk OMF file, not only
   the extracted LCONST image.
6. Add ExpressLoad full-file adapters for the exact code-image candidates.
